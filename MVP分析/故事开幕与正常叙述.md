## 故事开幕与正常叙述

此阶段在主角设定保存后启动。世界观、主角设定、已邀请的关键人物设定、地点、NPC、物品等持续输入给LLM。

**通用 API 参数设定：**

- `response_format: { "type": "json_object" }` —— 保证 API 返回合法 JSON 语法（DeepSeek 的 `json_object` 模式不支持传入自定义 schema，具体格式需通过 system prompt 传达）
- `temperature` 按阶段分层：开幕 0.7，叙述I 0.8，叙述II 0.7，历史总结 0.3

***

### 1. 进入故事开幕

用户点击对话区下方的 `<故事开幕>` 按钮。

\[系统锁定 `<故事开幕>` 按钮，使其不可再次点击。]

\[系统输出："故事即将开幕。"]

\[系统组装 LLM 输入]

#### 输出格式（故事开幕）

LLM 必须输出一个 JSON 对象，格式如下：

```json
{
  "narration": "开幕叙述文本",
  "locations": [
    { "name": "地点名", "description": "描述" }
  ],
  "npcs": [
    { "name": "NPC名", "description": "描述" }
  ],
  "items": [
    { "name": "物品名", "status": "已获得/已失去", "description": "描述" }
  ],
  "options": [
    "A. 选项内容",
    "B. 选项内容",
    "C. 选项内容",
    "D. 自由行动"
  ]
}
```

- `locations` / `npcs` / `items` 若无相应内容，设为空数组 `[]`
- `items` 中每个元素必须标明 `status`（已获得 / 已失去）
- NPC 需为新出场人物或有重要状态更新的旧人物
- 物品需为主角已获得的新物品，或失去的旧物品
- `options` 必须恰好包含 4 个元素，以 A. B. C. D. 开头，最后一个固定为 `"D. 自由行动"`

此 JSON Schema 通过 `system` prompt 文本直接告知 LLM（简称 `{STORY_OPENING_SCHEMA}`）。

#### LLM 输入 Schema（故事开幕）

| roles 参数                 | 内容                                                         |
| :----------------------- | :--------------------------------------------------------- |
| `messages[0]` (`system`) | 系统辅助性指令 + 完整 `{STORY_OPENING_SCHEMA}` JSON 格式文本 |
| `messages[1]` (`user`)   | 世界观描述全文 + 主角设定全文                                           |

> **Note – API 实际请求的 messages 示例：**
>
> ```json
> [
>   {
>     "role": "system",
>     "content": "我在尝试一种新型的AI跑团……现在，请为我撰写一个符合设定、有代入感的跑团故事开幕。\n\n你的输出必须是一个合法的 JSON 对象。请严格按照以下 JSON 格式输出，不要输出任何 JSON 以外的内容（不要用 markdown 代码块包裹）：\n\n{  \"narration\": \"开幕叙述文本\",  \"locations\": [{ \"name\": \"地点名\", \"description\": \"描述\" }],  ……  \"options\": [\"A. 选项内容\", …, \"D. 自由行动\"]  }\n说明：- locations / npcs / items 若无相应内容，设为空数组 []……"
>   },
>   {
>     "role": "user",
>     "content": "世界观与人物设定如下：\n世界观描述：蒸汽朋克与古老魔法并存，科技与信仰冲突剧烈的大陆……\n主角设定：姓名：赵铁生，年龄：42岁，职业：退伍军人，现为码头工人……"
>   }
> ]
> ```

\[系统调用 LLM API，输入内容。]

\[系统解析 LLM 返回的 JSON 输出：]

- 验证 JSON 合法，检测 `narration` 字段存在。
- 将 `narration` 字段文本存入数据库 `chat_record` 字段。
- 将 `locations` 数组逐个解析，追加存入数据库 `location` 字段，同时填入UI右侧 `<重要地点>` 模块，同名直接覆盖。
- 将 `npcs` 数组逐个解析，追加存入数据库 `npc` 字段，同时填入UI右侧 `<重要npc>` 模块，同名直接覆盖。
- 将 `items` 数组逐个解析，追加存入数据库 `inventory` 字段，同时填入UI右侧 `<物品仓库>` 模块，同名直接覆盖。
- 将 `options` 数组序列化为文本，存入数据库 `option_buffer` 字段，覆盖原有内容。

\[系统将 LLM 本次输出的 `narration` 内容显示在对话框 KP 气泡。]

\[系统渲染 `locations`/`npcs`/`items` 在气泡底部（金色分割线之后）。]

\[系统在用户输入区上方渲染选项按钮：`<A> <B> <C> <D>`，仅显示字母。]

\[用户点击选项按钮的行为规则：]

- 点击任意按钮，输入区自动填入"选项X"（X为对应字母）。
- 允许点击多个按钮，如依次点击A和B，输入区显示"选项A和B"。
- 再次点击已选中按钮可取消，输入区内容实时刷新。
- D选项允许用户在"选项D"后继续输入自由文字描述，提示允许基于A/B/C修改
- 用户可在输入区直接手动修改选项文字，以最后一次发送内容为准。

\[用户点击对话发送按钮后，系统进入正常叙述循环。]

**Note: 后续可引入"召集小队成员"的固定选项。**

### 2. 正常叙述阶段

\[系统将用户输入的"选项X"替换为选项对应的具体文本内容，与用户附加的自由文字（如有）拼接，作为本轮 `用户当前prompt`，并存入数据库 `chat_record` 字段。]

\[系统组装LLM输入。]

#### 输出格式（叙述I / 叙述II 共用）

LLM 必须输出一个 JSON 对象。根据剧情是否需要投掷判定，选择格式 A 或 格式 B：

**格式 A —— 不需要投掷判定，正常叙事：**

```json
{
  "narration": "叙述文本",
  "locations": [
    { "name": "地点名", "description": "描述" }
  ],
  "npcs": [
    { "name": "NPC名", "description": "描述" }
  ],
  "items": [
    { "name": "物品名", "status": "已获得/已失去", "description": "描述" }
  ],
  "hp": 12,
  "san": 55,
  "options": [
    "A. 选项内容",
    "B. 选项内容",
    "C. 选项内容",
    "D. 自由行动"
  ]
}
```

- `hp` / `san` 若无变化则设为 `null`
- `locations` / `npcs` / `items` 若本轮无新内容，设为空数组 `[]`
- 不要为了填充而编造不存在的地点/NPC/物品
- `options` 必须恰好包含 4 个元素，最后一个固定为 `"D. 自由行动"`

**格式 B —— 需要投掷判定：**

```json
{
  "narration": "叙述文本（需要判定的地方用【】标注，如何判定写在"dice"对象中,可多个）"
  "dice": {
    "skill_name": "技能名（技能点）",
    "notation": "1d100/1d4 etc.",
    "success rate": "成功率"
  }
}
```

- 选择格式B时，`narration` 中仍需用【】包裹判定说明
- `dice` 字段必须包含 `skill_name`、`notation`三个子字段
- 格式B不包含 `locations`/`npcs`/`items`/`hp`/`san`/`options` 字段

此 JSON Schema 通过 `system` prompt 文本直接告知 LLM（简称 `{NARRATIVE_SCHEMA}`，含格式A和格式B）。

#### LLM 输入 Schema（叙述I —— 等待判定或叙事）

| roles 参数                      | 内容                                                       |
| :---------------------------- | :------------------------------------------------------- |
| `messages[0]` (`system`)      | 系统辅助性指令（含格式选择规则）+ 完整 `{NARRATIVE_SCHEMA}` JSON 格式文本 |
| `messages[1]` (`user`)        | 世界观描述全文 + 主角设定全文                                         |
| `messages[2..N-2]`            | 历史对话——按 `assistant` / `user` 交替排列                        |
| `messages[N-2]` (`assistant`) | 本轮 option\_buffer 内容（从 DB 读取，与当前轮次选项拼接）                  |
| `messages[N-1]` (`user`)      | 用户当前 prompt（本次行为）                                        |

> **Note – 叙述I API 实际请求的 messages 示例（P1 实现后）：**
>
> ```json
> [
>   {
>     "role": "system",
>     "content": "我在尝试一种新型的AI跑团……现在，你作为KP，需要根据玩家行为推进剧情。\n\n若剧情中需要进行技能/属性投掷判定，请选择格式 B……\n\n你的输出必须是一个合法的 JSON 对象。请严格按照以下 JSON 格式输出，不要输出任何 JSON 以外的内容（不要用 markdown 代码块包裹）：\n\n格式 A —— 不需要投掷判定……\n格式 B —— 需要投掷判定……\n说明：……"
>   },
>   {
>     "role": "user",
>     "content": "世界观设定：蒸汽朋克与古老魔法并存……\n主角设定：赵铁生，42岁，退伍军人……"
>   },
>   {
>     "role": "assistant",
>     "content": "（上轮 KP 完整输出的 narration 文本）"
>   },
>   {
>     "role": "assistant",
>     "content": "A. 走向那个深色风衣男人，试探他的身份\nB. 去码头管理员办公室报到\nC. 先在附近的酒馆歇歇脚，打听消息\nD. 自由行动"
>   },
>   {
>     "role": "user",
>     "content": "选项A"
>   }
> ]
> ```

\[系统调用 LLM API。]

\[系统解析 LLM 返回的 JSON。]

\[系统检测 JSON 中是否包含 `dice` 字段。]

#### 分支A：无 `dice` 字段

\[系统执行以下处理：]

- 将 `narration` 字段文本存入数据库 `chat_record` 字段。
- 将 `locations`、`npcs`、`items` 数组分别解析，追加存入对应数据库字段，并更新UI侧边栏模块。
- 若有 `hp` 或 `san` 字段（值不为 null），解析数值并在数据库/UI更新主角对应属性。
- 将 `options` 数组存入数据库 `option_buffer` 字段，覆盖原有内容。

\[系统将 narration 显示在对话框，渲染选项按钮 `<A> <B> <C> <D>`。]

\[等待用户输入，用户发送后系统按"叙述I"格式重新组装输入。]

***

#### 分支B：有 `dice` 字段

\[系统将 narration 文本显示在对话框（含【】判定说明），暂时禁用用户对话框输入权限]

\[系统在对话界面自动发送："确定使用该技能进行投掷判定吗？"，然后显示`<确定>`和`<取消并回退至上次行为>`两个按钮；若用户点击"取消"，系统将对话框状态回退至用户上次做出的行为，并清除相应缓存与LLM输出显示，比如回退到用户上次输入选项或点击"确定"的状态；若用户点击"确定"，系统执行以下指令：]

\[系统将 narration 字段文本和 dice 信息存入数据库 `chat_record` 字段。]
\[系统解析 `dice.notation` 中的骰子指令，按顺序调用本地真随机数生成程序，记录每次投掷的结果数字。]

\[系统将投掷结果以独立消息形式显示在对话框，格式为："【系统投掷结果】50, 30"。]

\[显示投掷结果期间，暂时禁用用户输入权限，直至LLM输出下一个无 `dice` 字段的文段]

\[系统自动按"叙述II"格式组装LLM输入。]

### 3. 对话历史总结机制

触发时机：系统将最新LLM输出存入 `chat_record` 字段之后，用户输入下一轮prompt之前。

[触发期间，系统禁止用户输入，并在对话区显示提示："系统正在更新记忆，请稍等片刻。"]

触发条件：`chat_record` 中除最新两条记录外的历史内容，预估总字数超过系统设定阈值。

[系统暂停正常叙述，向 LLM 发送总结指令。]

#### 输出格式（历史总结）

LLM 必须输出一个 JSON 对象，格式如下：

```json
{
  "summary": "总结文本"
}
```

此 JSON Schema 通过 `system` prompt 文本直接告知 LLM（简称 `{SUMMARY_SCHEMA}`）。

#### LLM 输入 Schema（历史总结）

| roles 参数                 | 内容                                                         |
| :----------------------- | :--------------------------------------------------------- |
| `messages[0]` (`system`) | 系统总结指令 + 完整 `{SUMMARY_SCHEMA}` JSON 格式文本    |
| `messages[1]` (`user`)   | 必要设定（世界观、主角、已解锁地点/NPC/物品）                           |
| `messages[2..N-1]`       | 历史对话记录——将 `chat_record` 中除最新两条外的所有条目逐条展开为独立 `assistant` / `user` messages |

系统调用时额外设置：

- `max_tokens: 800`（总结不用太长）
- `temperature: 0.3`（总结需精准）

> **Note – chat_record 顺序规则与 messages 映射：**
>
> `chat_record` 始终保持固定的交替模式：**以 assistant 开头，以 assistant 结尾**。
>
> ```
> chat_record = [
>   { role: "assistant", content: "开幕叙述..." },      // 开幕
>   { role: "user",      content: "选项A" },            // 玩家行为
>   { role: "assistant", content: "KP叙述..." },         // KP回应
>   { role: "user",      content: "选项B" },            // 玩家行为
>   { role: "assistant", content: "KP含dice叙述" },      // KP回应（含dice）
>   { role: "user",      content: "投掷结果：50,30" },   // 系统投掷结果→视为user
>   { role: "assistant", content: "NARRATION_II回应" },  // KP对投掷结果的叙述
>   ...
> ]
> ```
>
> 关键规则：
> - 开幕 narration 是第一条，role = assistant
> - 玩家每轮行为（选项）追加为 user
> - KP 每轮回应追加为 assistant
> - 系统自动发送的投掷结果追加为 **user**——因为从 API 视角看，它是 KP 需要回应的输入
> - 循环始终以 assistant 结尾，不会出现连续两个 user 或连续两个 assistant
> - 若历史已含往期 summary，它作为 assistant 类型插入最前端，不破坏交替规律
>
> **未来 Tool Calling 兼容性：**
> - 若后续用 function calling 替代 `<dice>` 标签，`chat_record` 中需新增 `role: "tool"` 类型
> - tool 调用流程：`user → assistant(tool_calls) → tool(result) → assistant(narration)`
> - tool 消息插入在两段 assistant 之间，**形式上打破了纯 alternating 模式**，但这是 API 原生支持的合法链
> - 总结/拼接时按 role 如实映射到 messages 即可，不需要特殊处理
> - 总结替换仍从最前一个 assistant 覆盖到最后一个 assistant 之前

> **Note – 历史总结 API 实际请求的 messages 示例：**
>
> 假设此时已有多轮对话，chat_record 中除最新两条外需要总结：
>
> ```json
> [
>   {
>     "role": "system",
>     "content": "你是跑团KP，现在需要帮我总结迄今剧情。你给出的总结要保证自己后续可以通过该总结正常推进跑团进程，保证故事的合理性，暗示故事可能的伏笔。\n\n你的输出必须是一个合法的 JSON 对象。请严格按照以下 JSON 格式输出，不要输出任何 JSON 以外的内容（不要用 markdown 代码块包裹）：\n\n{  \"summary\": \"总结文本，300字以内\"  }"
>   },
>   {
>     "role": "user",
>     "content": "故事必要设定如下：\n世界观：蒸汽朋克与古老魔法并存，科技与信仰冲突剧烈的大陆……\n主角：赵铁生，42岁，退伍军人，码头工人……\n已解锁地点：海港城码头区、老城区酒馆……\n已知NPC：老乞丐，深色风衣男人……\n主角物品栏：一把生锈的匕首"
>   },
>   {
>     "role": "assistant",
>     "content": "夜幕低垂，码头区弥漫着咸腥的海风。赵铁生拖着疲惫的身躯走下货船……（开幕叙述全文）"
>   },
>   {
>     "role": "user",
>     "content": "选项A：我环顾四周，看看有没有什么可疑的人。"
>   },
>   {
>     "role": "assistant",
>     "content": "你借着码头昏黄的瓦斯灯环视四周。远处栈桥边，一个深色风衣的瘦高男人倚靠在木箱旁。老乞丐蜷缩在仓库门边。"
>   },
>   {
>     "role": "user",
>     "content": "选项A：走向那个风衣男人，试探他的身份。"
>   }
> ]
> ```
>
> （注：上例中 chat_record 最后两条为 "assistant(最新KP输出)" + "user(最新玩家行为)"，这两条被保留不参与总结；示例展示的是参与总结的更早记录。）

[系统接收到含 `summary` 字段的 JSON 输出后，执行记忆替换：]

- 保留 `chat_record` 中最新两条记录（即最近一轮的 user prompt + assistant 输出），删除其余所有旧记录。
- 将 `summary` 字段内的总结文本以 `{ role: "assistant", content: summary文本 }` 格式插入到 `chat_record` 的最前端。

[替换完成后，系统恢复用户输入权限，并在对话区显示提示："系统已完成自动记忆更新。"]

若此时处于投掷判定流程中（即系统刚返回投掷结果，等待向LLM输入时触发了总结），总结完成后系统自动继续发送判定结果，流程不受中断。

### 4. JSON 输出保底机制

每次 LLM 输出后，系统尝试 `JSON.parse()` 解析。

对于故事开幕/叙述I/叙述II 阶段：

- **若解析失败或缺少** **`narration`** **字段**：自动重试一次，在 `system` 消息末尾追加 `【请一定只输出合法的 JSON，且必须包含 "narration" 字段】`。
- **若重试后仍失败**：使用 LLM 完整输出原文作为备用，按 `type: 'raw'` 存入 `chat_record`；系统提示用户"LLM 两次均未输出合法 JSON，已使用完整输出作为备用。"

对于历史总结阶段：

- **若解析失败或缺少** **`summary`** **字段**：采用相同流程：先重试注入提醒，若仍缺失则使用完整 LLM 输出作为总结文本。

***

### 前端设计总结

- `<故事开幕>` 按钮：仅可点击一次，点击后锁定变灰。
- 叙事区：流式展示 `narration` 字段文本，渲染为沉浸式 KP 气泡。
  - 气泡底部：金色分割线后渲染 `locations`/`npcs`/`items`（小号无衬线字体）
  - 再一条金色分割线后渲染 `options` 内容（金色文字）
- 选项区：自动从 `options` 数组生成可点击按钮 `<A> <B> <C> <D>`（仅需显示字母），互动效果参前。
- 侧边栏模块：实时展示已解锁地点、已知NPC、物品仓库的更新，内容从数据库读取。
- 投掷结果：以独立系统消息形式展示在对话区。

### 数据库设计总结

| 字段               | 用途                     | 更新方式                       |
| :--------------- | :--------------------- | :------------------------- |
| `chat_record`    | 存储对话历史（LLM输出与用户prompt） | 每轮追加 narration 文本；总结时替换旧内容 |
| `option_buffer`  | 暂存当前轮次选项文本             | 每轮LLM输出后从 options 数组提取并覆盖  |
| `location`       | 已解锁重要地点列表              | LLM输出含 locations 数组时逐个追加   |
| `npc`            | 已知NPC列表及状态             | LLM输出含 npcs 数组时逐个追加或更新     |
| `inventory`      | 主角物品仓库                 | LLM输出含 items 数组时逐个追加或更新    |
| `protagonist`    | 主角完整设定（含技能）            | 人物设定阶段写入，可手动修改             |
| `world_settings` | 世界观设定                  | 世界设定阶段写入，理论上不再修改           |
