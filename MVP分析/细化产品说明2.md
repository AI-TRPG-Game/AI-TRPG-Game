## 1) 你现在这套产品的“核心约束”再确认（不变的底层原则）

1. **LLM 只负责：读输入 → 生成文本/结构化输出**  
2. **系统负责：**
   - 决定给 LLM 输入什么（拼装上下文）
   - 决定 LLM 输出要长什么样（范式/校验/重试）
   - 执行非语言的“硬规则”（骰子、状态判定、属性变化）
   - 把“可长期记住”的要点存成结构化数据（世界观/人物/任务/物品等）
3. **前端是“控制台”**：玩家既在聊天，也在编辑结构化设定；发送键是“系统提交一次回合”的明确触发器。

这几条决定：你最终要的不是聊天框，而是一个 **“编排器 Orchestrator + 状态库 WorldState + 输出解析器 Parser”** 驱动的交互系统。

## 2) 系统模块：从“概念”到“工程可落地”的分层

你在《细化产品说明》里列的 7 个分区非常对，我这里把它们改写成“最小职责 + 输入输出边界”（后面写代码会直接按这些边界落类/模块）。

### 2.1 Game Orchestrator（编排器 / 总控）
- **职责**：处理用户点击/发送产生的“事件”，选择走哪个流程（世界设定、人物设定、开幕、normal 回合、鉴定回合等），并协调其余模块。
- **输入**：UI事件（点击某按钮、提交文本、保存设定、选择选项…）
- **输出**：给 UI 的渲染包（叙事文本、选项、可存档 NPC、要更新的状态字段等）

### 2.2 Mode Manager（模式管理）
- **职责**：维护当前模式（Meta/Normal；未来可扩展 Setup/Opening/Check 等“子模式”）
- **关键点**：你文档里其实不仅是 Meta/Normal，还隐含了“阶段”（世界设定阶段、开幕阶段、鉴定阶段）。建议后续做成：
  - `Mode: meta | normal`
  - `Phase: setup_world | setup_pc | opening | playing | checking`
  这样规则更清晰。

### 2.3 World State DB（特殊数据库 / 世界状态）
即你强调的“和叙事历史分离”的结构化信息。
- **内容**（最小集）：
  - `world_background`（世界观/背景）
  - `pc`（主角卡）
  - `npcs[]`（已存档配角卡）
  - `quest_core`（核心任务）
  - `quest_current`（当前任务）
  - `inventory[]`（关键物品）
  - `rulesets`（输出范式、人物卡格式、开幕格式等“模板规则”）
  - `dice_log[]`
- **注意**：你文档要求“用户保存后系统自动发送给 LLM”，所以它必须是**可随时取用的“权威状态源”**。

### 2.4 Context Manager（记忆/上下文管理）
你现在说“简化版历史管理：滑动窗口 + 摘要/索引”。  
所以它至少要能输出两段内容：
- `short_term_messages`：最近 N 轮（用于局部连贯）
- `long_term_summary`：更早内容压缩摘要（用于长期一致）
并对输入装配器提供一个统一接口：`getContext(sessionId, tokenBudget)`。

> 你现在做的是“纯聊天无记忆 + 固定选项范式”，属于早期阶段；但从你文档推演的最终体验看，ContextManager 是必须的，只是可以后置。

### 2.5 Input Assembler（输入装配器）
你文档里反复出现 “系统将向 LLM 发送：1世界观 2人物设定 3历史对话 4用户prompt …”。  
Input Assembler 就是把这些拼成最终 messages 的工人。

最关键是：它要按“流程”拼装：
- 世界设定生成用的 prompt
- 人物卡生成用的 prompt（带 format）
- 开幕 prompt（带 opening format）
- normal 回合 prompt（带 normal output format）
- checking 回合 prompt（带 check format）

### 2.6 Output Parser & Validator（输出解析与校验）
- **解析**：从 LLM 输出中提取
  - 叙事文本
  - A/B/C/D 选项（你现在已实现轻量版）
  - 重要 NPC（可点击存档）
  - check 阶段需要的“是否检定/检定类型/骰子参数”
- **校验**：不符合则触发重试（你在截图里提到的 OutputValidator）
  - 早期：校验 A/B/C/D 末尾范式即可
  - 后期：升级为 JSON schema 校验（叙事 + options + npc_updates + state_updates）

### 2.7 Rule Engine（外部规则）
- 真随机掷骰（明骰/暗骰）
- 属性阈值触发状态变化（例如 san < x -> 疯狂提示）
- 将结果写入 dice_log，并作为“事实”提供给 LLM（LLM 不允许编骰子）

## 3) “完整但简化”的用户游玩过程模拟（按你的文档重走一遍）

下面这段是我按你的“用户情景模拟”做的**时序回放**，并在每一步标注哪些模块会被调用、读写什么数据。

---

### 阶段 0：用户进入产品 / 创建新存档
**用户行为**：打开网页 → 点击“新建存档”  
**系统行为**：
- Orchestrator：创建 session
- WorldStateDB：初始化空白 world/pc/quest/inventory/rulesets
- ModeManager：默认 `mode=meta` 或 `phase=setup_world`（你文档倾向“设定阶段是 meta 且与 normal 分开”）

**UI**：出现“故事设定”入口（世界观、人物、开幕按钮等）

---

### 阶段 1：世界观与背景设定（Meta / Setup World）
**用户行为**：点击“开启故事设定”，输入关键词：“偏远小镇、废弃教堂、克苏鲁”  
**系统行为**：
- ModeManager：`mode=meta, phase=setup_world`
- InputAssembler：装配一个“生成世界观/背景”的指令（可有固定模板）
- LLM：返回世界观文本（此时可以不强制 JSON，只要结构清晰）
- UI：展示文本，用户满意后把内容填到右侧“世界观与背景”框，点击“确定保存”
- WorldStateDB：写入 `world_background`
- Orchestrator：可选自动发一条“已保存”回显给聊天区

**关键设计点**：  
这里你坚持“用户手动确认/复制/修改”是对的——能避免 LLM 乱改关键设定，也降低系统误写数据库的风险。

---

### 阶段 2：主角人设设定（Meta / Setup PC）
**用户行为**：点击“开启设定主角”，输入：“女调查记者，性格冷静，童年阴影…”  
**系统行为**：
- ModeManager：`mode=meta, phase=setup_pc`
- InputAssembler：拼装给 LLM：
  1) `world_background`（如已存在）
  2) `pc_card_format`（人物卡字段要求：姓名/年龄/性别/…）
  3)（可选）用户之前对主角的修改记录
  4) 当前用户输入
- LLM：输出“主角卡（自然语言或半结构化）”
- 用户反复调整 prompt
- 用户点击“存档当前人物设定”
- OutputParser：把 LLM 输出解析为结构化 pc（哪怕是 key-value 文本也要能提取）
- WorldStateDB：覆盖写入 `pc`
- UI：右侧“主角设定”面板更新，用户可再手工编辑并保存（再写回 DB）

**关键设计点**：  
你的 note “存档会完全覆盖上次内容”非常合理，避免增量合并的复杂度（后期再做 diff）。

---

### 阶段 3：故事开幕（Normal / Opening）
**用户行为**：点击“故事开幕”  
**系统行为**：
- ModeManager：切换 `mode=normal, phase=opening`
- InputAssembler：拼装给 LLM：
  - world_background + pc + opening_format（开幕输出格式）
- LLM：输出开幕叙事 + 核心任务引入
- UI：显示开幕叙事，并给一个固定按钮“接收任务”
- 用户点击“接收任务”
- Orchestrator：进入 `phase=playing`，开始 normal narrative 的循环

**关键设计点**：  
这里“开幕输出格式”建议你固定模板（比自由聊天更像产品）。

---

### 阶段 4：故事正常推进（Normal / Playing 循环）
这是主循环。你文档里强调：“**所有 UI 操作最终以点击发送为准，系统才整合 input 并发送给 LLM**”。  
这其实是在定义一个**回合提交模型**。

#### 4.1 用户做动作（可能来自三种来源）
- 点击选项 A/B/C/D（系统把选项写进输入框，允许用户追加文字）
- 在输入框自由输入
- 点击 NPC 按钮（“存档该 NPC”）或修改右侧状态（meta 数据变更）

最终统一在“发送”时提交。

#### 4.2 系统处理一次回合的两段式（你文档提到的“鉴定管线”）
你设计的是：

1) **先鉴定：判断是否需要判定**
- InputAssembler 装配一个“鉴定请求 prompt”（用 check_format）
- LLM 输出：“需要/不需要检定 + 检定类型/骰子参数”
- OutputParser 解析检定需求

2) **若需要：RuleEngine 掷骰并产出结果**
- RuleEngine 产生随机点数
- 写入 dice_log
- 把“检定结果”作为事实加入下一次输入

3) **再推进剧情：normal narrative prompt**
- InputAssembler 装配 normal prompt（meta 状态 + 历史摘要 + 用户动作 + 骰子结果 + normal 输出格式）
- LLM 输出：叙事 + NPC/物品高亮 + A/B/C/D
- OutputParser：提取
  - 叙事文本
  - A/B/C/D
  - 新出现的重要 NPC（供用户点击存档）
  - 关键物品（供用户存档）
- Orchestrator：更新 WorldStateDB（若用户点击“存档 NPC/物品”）

**关键设计点**：  
你这里选择“是否需要检定由 LLM 判断”是合理的（节省你写大量规则判定），但必须通过固定格式让系统可靠解析；而最终结果仍由系统随机生成，保持公平性。

## 4) 代码如何实现：从你现在的 Demo 过渡到最终架构的“最小演进路径”

你现在已经有：
- provider 调用（DeepSeek/OpenAI-compatible）
- 简单 chat
- A/B/C/D 选项解析与按钮回传（在 simple chat 模式）

接下来如果按你“叙事系统具体逻辑”推进，我建议**按阶段抽象接口，而不是一下子写完 7 大模块**。最稳的演进路径是：

### 第一步（你正在做）：格式与交互闭环
- 输出范式：A/B/C/D（D 自由活动）
- UI：按钮渲染 + 点击自动发送
- 校验：失败降级 / 失败重试一次（你之前问的 A/B/C）

你已经达成了 80%。

### 第二步：把“故事设定”从聊天里拆出来（Meta 面板流程化）
增加 UI 按钮：
- “开启故事设定（世界观）”
- “开启人物设定（主角）”
- “故事开幕”
并让 Orchestrator 在不同按钮下使用不同的 system prompt 模板（InputAssembler 的雏形）。

### 第三步：引入“回合提交模型”
- 选项点击不是直接发送，而是写入输入框（允许追加）
- 用户点击发送时，Orchestrator 汇总：
  - 当前输入框文本
  - 是否选了 A/B/C/D（可多选）
  - 是否存档了 NPC/物品（点击行为）
  - 是否修改了 world/pc
再一次性提交。

### 第四步：引入鉴定两段式管线（Checking Phase）
- 一个非常小的 check_format（例如：输出 `needs_check: yes/no` + `dice: d20` + `reason`）
- 系统掷骰
- 再 normal prompt 推进

到这一步，你的“叙事系统”才真的成立。

## 5) 我理解你文档里的“风险点”（提前标出来，后面写代码会避坑）

1) **让用户手动确认再写 DB**是对的；全自动抽取写 DB 会很容易写错，导致剧情崩坏。  
2) **“点击发送才提交回合”**是个非常关键的交互约束，建议从现在就贯彻（否则以后加复杂 UI 会很难收束）。  
3) **解析的可靠性**：A/B/C/D 是一个很好的“低成本结构化过渡”；之后再上 JSON schema。  
4) **模式不止 meta/normal**：你其实还需要 phase（setup/opening/checking/playing），否则后面 prompt 组合会混乱。  
5) **纯前端 A 方案的限制**：Key 安全/CORS/限流都不稳；你们后期走向 B 是必然的，但可以等玩法验证后再做。
