## 人物设定阶段

此阶段承接世界设定完成后，玩家为故事创建主角。系统保持与上一阶段相同的简化逻辑：历史记录仅保存用户手动输入与LLM输出，不涉及meta/normal模式切换，关键设定在触发历史总结指令时才发送给LLM。

**通用 API 参数设定：**

- `response_format: { "type": "json_object" }` —— 保证 API 返回合法 JSON 语法（DeepSeek 的 `json_object` 模式不支持传入自定义 schema，具体格式需通过 system prompt 传达）
- `temperature: 0.2` —— 数值计算需极低随机性

---

### 1. 进入人物设定

用户点击对话区下方的 `<开启人物设定>` 按钮。

[系统输出：人物设定阶段。]

[系统在前端对话区输出一条固定指引，无需调用 LLM]：

> "接下里请为你的故事设定主角。你可以直接描述，比如'一个退伍老兵，沉默寡言但心地善良'，或'一个好奇心过重的医学院学生'。我会为你生成一份详细的人物档案。调整满意后请点击'保存主角设定'按钮。"

此时对话区仅显示上述指引，等待用户输入。

---

### 2. 初次生成主角设定

用户输入关键词描述，例如："一个退伍老兵，沉默寡言但心地善良，40岁左右"，并按下发送。

[系统组装 LLM 输入]

#### 输出格式（人物设定）

LLM 必须输出一个 JSON 对象，格式如下：

```json
{
  "character_card": {
    "name": "姓名",
    "age": 42,
    "gender": "性别",
    "occupation": "职业",
    "personality": "性格描述",
    "portrait": "人物肖像与重要经历",
    "attributes": {
      "strength": 50,
      "dexterity": 50,
      "constitution": 60,
      "size": 55,
      "appearance": 40,
      "intelligence": 60,
      "willpower": 65,
      "education": 80
    },
    "hp": 11,
    "san": 65,
    "credit_rating": 20,
    "occupational_skills": [
      { "name": "技能名", "value": 60 },
      { "name": "技能名", "value": 50 }
    ],
    "personal_skills": [
      { "name": "技能名", "value": 40 },
      { "name": "技能名", "value": 30 }
    ],
    "inventory": ["物品1", "物品2"]
  }
}
```

其中：
- `occupational_skills` 为 8 个本职技能
- `personal_skills` 为 4 个非本职技能
- 每个技能元素包含 `name`（技能名）和 `value`（点数）

以下为数值计算规则（在 `system` 消息中说明）：

1. 根据用户描述/世界观/CoC7th规则合理补全，属性与技能要符合职业等人物设定
2. 8大属性点总和为460，单项属性在15-90之间
3. HP=（体质+体型）%10，SAN=意志
4. 各类技能（包括信用评级）的点数计算规则为基础+职业+兴趣，基础值固定
5. 职业技能点只能分配给本职技能和信用评级，兴趣技能点可分配给所有技能（信用评级除外）
6. 信用评级基础值为0，代表财富与社会地位，要符合职业设定，且占用职业技能点
7. 单项本职技能点数不超过80，非本职技能不超过50
8. 所有职业的基础数值由时代背景下普通人样貌决定，与角色本身无关；基础数值有4档：0为克苏鲁神话和信用评级，1为普通人一般接触不到的技能，10为普通人偶尔用到/一般熟悉的技能，25为生活技能或生物本能
9. 职业技能点总和=教育*4 或 教育*2+力量*2 等，根据职业决定
10. 兴趣技能点总和=智力*2
11. 核查数值计算，注意不要把技能基础点数计入职业技能点与兴趣技能点总和的限制

此 JSON Schema 通过 `system` prompt 文本直接告知 LLM（简称 `{CHARACTER_SCHEMA}`）。

#### LLM 输入 Schema

| roles 参数                 | 内容                                       |
| :----------------------- | :--------------------------------------- |
| `messages[0]` (`system`) | 系统辅助性指令（含世界观描述）+ 数值计算规则（上述11条）+ 完整 `{CHARACTER_SCHEMA}` JSON 格式文本 |
| `messages[1]` (`user`)   | 用户当前 prompt                              |

> **Note – API 实际请求的 messages 示例：**
>
> ```json
> [
>   {
>     "role": "system",
>     "content": "我在尝试一种新型的AI跑团……现在我们要创建主角的人物设定。\n\n世界观描述如下：蒸汽朋克与古老魔法并存……\n\n以下为数值计算规则：\n1. 根据用户描述/世界观/CoC7th规则合理补全……\n……\n11. 核查数值计算……\n\n你的输出必须是一个合法的 JSON 对象。请严格按照以下 JSON 格式输出，不要输出任何 JSON 以外的内容（不要用 markdown 代码块包裹）：\n\n{  \"character_card\": {  \"name\": \"姓名\",  \"age\": 42,  ……  }  }\n说明：\n- occupational_skills 为 8 个本职技能，personal_skills 为 4 个非本职技能"
>   },
>   {
>     "role": "user",
>     "content": "一个退伍老兵，沉默寡言但心地善良，40岁左右"
>   }
> ]
> ```

[系统调用 LLM API，stream 方式将生成内容展示在对话区]

[系统解析 LLM 返回的 JSON，验证 `character_card` 字段存在且包含所有必要子字段]

Note: AI在后续跑团对话中只需要记住8个本职技能和4个非本职技能，当需要使用超出这些技能范围外的技能进行判定时，按照时代背景常理生成技能基础数值并根据基础数值来判定即可；系统在存入主角设定时，在技能部分自动补充："其余技能：根据时代背景常理自动生成基础数值，需要时据此判定"

---

### 3. 调整与迭代

用户阅读后，可以在对话框中继续提出修改意见，如："背景故事太苦了，让他有个女儿吧，是他活下去的唯一动力。"

[系统重新组装输入，此次输入中包含之前的对话历史]

#### LLM 输入 Schema（调整轮次）

| roles 参数                 | 内容                                       |
| :----------------------- | :--------------------------------------- |
| `messages[0]` (`system`) | 系统指令 + 世界观 + 数值规则 + 完整 `{CHARACTER_SCHEMA}` JSON 格式文本（同初次） |
| `messages[1]` (`user`)   | 用户首次 prompt                              |
| `messages[2]` (`assistant`) | LLM 上次 JSON 输出原文                         |
| `messages[3]` (`user`)   | 用户最新修改意见                                 |
| ...                    | 可继续追加多轮                                |

> **Note – 两轮调整后的 messages 示例：**
>
> ```json
> [
>   {
>     "role": "system",
>     "content": "……（系统指令 + 世界观 + 数值规则 + {CHARACTER_SCHEMA}，同上）……"
>   },
>   {
>     "role": "user",
>     "content": "一个退伍老兵，沉默寡言但心地善良，40岁左右"
>   },
>   {
>     "role": "assistant",
>     "content": "{\"character_card\": {\"name\": \"赵铁生\", ...}}"
>   },
>   {
>     "role": "user",
>     "content": "背景故事太苦了，让他有个女儿吧，是他活下去的唯一动力。"
>   }
> ]
> ```

[LLM 生成新版本人物档案 JSON 并展示]

用户可多次重复此步骤，直至满意。

---

### 4. 存档主角设定

用户对当前对话中最后一条人物档案满意后，点击对话区下方的 `<保存主角设定>` 按钮。

[系统解析最新 assistant 消息中的 JSON，提取 `character_card` 对象，序列化为文本]

[系统将序列化后的文本自动填入UI右侧 `<主角设定>` 模块，已有则覆盖，并自动存入 `protagonist` 字段]

[系统在对话区提示："主角设定已保存，你可以在侧边栏随时查看，支持手动修改，并在点击`<保存修改>`后自动存档；如果不需要再调整，请点击 `<故事开幕>` ，开始冒险。"]

用户可以实时修改`<主角设定>`模块中的内容，系统在其点击位于模块下方的`<保存修改>`后覆盖写入数据库。

[同时，`Game Orchestrator` 会将该人物档案文本标记为当前会话的活跃设定。在后续对话中，重新输出给LLM，使其长期记忆该设定。]

---

### 5. JSON 输出保底机制

每次 LLM 输出后，系统尝试 `JSON.parse()` 解析。

- **若解析失败或缺少 `character_card` 字段**：自动重试一次，在 `system` 消息末尾追加 `【请一定只输出合法的 JSON，且必须包含 "character_card" 字段】`。
- **若重试后仍失败**：使用 LLM 完整输出原文作为备用，存入对话历史；系统提示用户"LLM 两次均未输出合法 JSON，已使用完整输出作为备用。"

---

### 前端设计总结

- `<开启人物设定>` 按钮：进入阶段，显示固定指引
- `<保存主角设定>` 按钮：提取 JSON 中 `character_card` 内容，覆盖写入数据库
- `<主角设定>` 存储模块：右侧面板显示，支持手动编辑
- `<保存修改>`按钮：位于`<主角设定>`的底部，点击后将主角设定覆盖写入数据库
- `<故事开幕>` 按钮：结束设定阶段，进入故事开幕

### 数据库设计总结

`protagonist` 字段：存储主角最新档案；系统在存入主角设定时，在技能部分自动补充："其余技能：根据时代背景常理自动生成基础数值，需要时据此判定"
