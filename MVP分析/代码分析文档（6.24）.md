# AI+TRPG 代码框架详细说明

## 一、整体架构

```
┌─────────────────────────────────┐     ┌──────────────────────────────────────┐
│        前端 (Vite SPA)           │     │            后端 (Node.js)              │
│                                  │     │                                       │
│  index.html         UI界面       │     │  api/GameController.js   REST路由层    │
│  src/main.js        应用入口     │◄───►│  api/StreamEmitter.js   SSE推送管理    │
│  src/ui/            界面控制器   │ SSE │                                       │
│  src/api/           HTTP客户端   │     │  orchestrator/GameOrchestrator  核心调度│
│  src/persistence/   IndexedDB   │     │  orchestrator/PhaseManager     阶段管理 │
│                                  │     │                                       │
│                                  │     │  services/  12个服务模块              │
│                                  │     │  domain/    GameSession + 枚举       │
│                                  │     │  llm/       LLM Provider             │
│                                  │     │  persistence/ SQLite数据层            │
└─────────────────────────────────┘     └────────────────┬─────────────────────┘
                                                         │
                                                         ▼
                                               ┌──────────────────┐
                                               │   LLM API        │
                                               │  (OpenAI兼容)     │
                                               └──────────────────┘
```

- **前端**：Vite 构建的单页应用，仅包含 UI 交互和 IndexedDB 会话缓存，不处理游戏逻辑
- **后端**：Express 无状态 API 服务，所有游戏逻辑都在后端，每次请求携带完整 session 快照
- **无状态设计**：后端不保存内存状态，每次请求通过 `session` 对象恢复完整上下文后处理，处理完毕通过 `SessionRepository` 写回 SQLite

## 三、FlowType 与 Prompt 体系

6 种 `FlowType` 对应 6 套不同的 system prompt + JSON Schema + 温度参数：

| FlowType | 触发时机 | 温度 | tokens | thinking | 验证字段 |
|:---|:---|:---|:---|:---|:---|
| `WORLD_GEN` | 世界设定阶段每次玩家输入 | 0.3 | 1024 | 否 | `world_description` |
| `CHARACTER_GEN` | 人物设定阶段每次玩家输入 | 0.2 | 4096 | 否 | `character_card` |
| `STORY_OPENING` | 故事开幕 | 0.7 | 2048 | 是 | `narration` |
| `NARRATION_I` | 故事阶段玩家输入 | 0.8 | 4096 | 是 | `narration` |
| `NARRATION_II` | 掷骰后延续叙事 | 0.7 | 4096 | 是 | `narration` |
| `HISTORY_SUMMARY` | 对话超过 8000 字符 | 0.3 | 800 | 否 | `summary` |

所有阶段的 Prompt 都遵循 CoC 7th 规则，叙事阶段支持两种输出格式：**格式 A（无骰子，含 locations/npcs/items/hp/san/options）** 和 **格式 B（有骰子，仅含 narration + dice 字段）**。

## 四、玩家一次常规游玩的完整体验闭环

以下用"玩家视角 → 前端行为 → 后端 API → 后端逻辑链 → LLM调用"的完整链路，展示从打开页面到持续游玩的全过程。

### 阶段 0：页面初始化

```
玩家打开 http://localhost:5173
  → index.html 渲染 UI
  → GameUIController._init()
      → 检查 URL hash / localStorage 中的 sessionId
      → 有存档：SessionStore.getSession(id) 从 IndexedDB 恢复
      → 无存档：ApiClient.createSession()
          → POST /api/sessions
          → 后端 GameOrchestrator.createSession() → SessionRepository.create() → SQLite 写入
          → 返回 { session }
        → 随后自动 ApiClient.enterWorldSetting(session)
          → POST /api/sessions/:id/enter-world
          → 后端设置 phase=WORLD_SETTING, subState=AWAITING_INPUT
          → 前端显示引导语："请用几句话描述你希望的世界类型..."
      → GameUIController._restoreUI()
          → 遍历 displayLog[] 渲染历史消息
```

### 阶段 1：世界设定 (WORLD_SETTING)

```
玩家输入："一个维多利亚时代伦敦背景的克苏鲁风格侦探故事"
  → _sendMessage() 
      → 前端追加 user 气泡、锁定输入、显示等待提示
      → ApiClient.sendMessage(session, text, handlers)
          → POST /api/sessions/:id/message  → 拿到 streamId
          → 建立 SSE 连接 GET /api/streams/:streamId
          
  → 后端 GameOrchestrator.handleMessage()
      → PhaseManager.canPerformAction() ✓ 校验通过
      → subState = LLM_STREAMING → 存档
      → 写入 setupHistory.world: {role:'player', content:'...'}
      → PhaseManager.getFlowType() → WORLD_GEN
      → _runLlmFlow(session, WORLD_GEN, text, streamId)
          │
          ├─ InputAssembler.assemble(WORLD_GEN, session)
          │     → system message = WORLD_INSTRUCTION (CoC7th规则 + JSON Schema)
          │     → messages 填充 setupHistory.world 历史
          │     → 返回 { messages, temperature:0.3, maxTokens:1024 }
          │
          ├─ _callLLMWithRetry(session, assembled, WORLD_GEN, streamId)
          │     → 发送 debug_prompt 事件(可选, God's Eye)
          │     → OpenAICompatibleProvider.generateStream()
          │         → POST DeepSeek API, stream=true, response_format:json_object
          │         → 返回完整 JSON 文本（不在流式中推 chunk）
          │     → 发送 debug_raw 事件
          │     → jsonOutputParser.parse(raw) → 解析 JSON
          │     → 验证 world_description 字段存在
          │     → 若缺失：自动重试一次（在 user message 末追加提醒）
          │     → 若仍失败：raw 兜底
          │     → textRefiner.refine(WORLD_GEN, parsed)
          │         → 提取 world_description 文本 → { plainText, html }
          │     → _streamSimulated(plainText, streamId)
          │         → 每5字符 / 20ms 间隔 emit chunk 事件（打字机效果）
          │     → emit llm_complete 事件（html 最终渲染）
          │
          ├─ OutputProcessor.process(WORLD_GEN, session, parsed, raw)
          │     → EntityUpdater.applySetupHistory() → 写入 setupHistory.world
          │     → 分支：'SETUP' → 返回 { branch:'SETUP', raw }
          │
          └─ 保存 session, subState=AWAITING_INPUT
          
  → 前端 SSE 接收:
      chunk → _botText 逐字累加渲染（打字机效果）
      llm_complete → 替换为最终 HTML（带金色分割线）
      input_lock:false → 解锁输入框
      done → 保存到 IndexedDB, 更新侧边栏
```

### 阶段 2：存档世界观 → 进入人物设定

```
玩家点击"存档当前世界观"
  → _saveWorld()
      → POST /api/sessions/:id/save-world
  → 后端 GameOrchestrator.saveWorld()
      → PhaseManager.canPerformAction(SAVE_WORLD) ✓
      → SaveExtractor.getLatestKpOutput(session, 'world')
           → 从 setupHistory.world 倒序找最新 KP 输出
      → SaveExtractor.extractWorldFromRaw(raw)
           → JSON 解析 → 提取 world_description 字段
      → session.worldSettings = 世界观文本
      → 前端显示"世界观已保存"

玩家点击"开启人物设定"
  → _enterCharacter()
      → POST /api/sessions/:id/enter-character
  → 后端 GameOrchestrator.enterCharacterSetting()
      → PhaseManager.canPerformAction(ENTER_CHARACTER_SETTING) ✓
         （前置条件：worldSettings 非空、phase=WORLD_SETTING）
      → advancePhase: phase → CHARACTER_SETTING
      → subState = AWAITING_INPUT
      → 前端显示人物设定引导语
```

### 阶段 3：人物设定 (CHARACTER_SETTING)

```
玩家输入："一位30岁左右的退役侦探，擅长观察..."
  → 与阶段1相同的 sendMessage 流程
  → FlowType → CHARACTER_GEN
  → InputAssembler:
      → 将世界观描述注入 system message 开头
      → 填充 setupHistory.character 历史
  → LLM 返回 character_card JSON (name/age/gender/occupation/personality/
      portrait/attributes/hp/san/skills/inventory)
  → TextRefiner._refineCharacter() 生成可读文本
  → 玩家可以多轮对话调整人物设定

玩家点击"保存主角设定"
  → _saveCharacter()
      → POST /api/sessions/:id/save-character
  → 后端 SaveExtractor.extractCharacterFromRaw(raw)
      → 将 character_card JSON 序列化为固定格式文本
      → 合并 "其余技能：根据时代背景常理自动生成基础数值..."
  → session.protagonist = 序列化文本
  → 前端主角侧边栏更新

玩家点击"故事开幕"
  → _openStory()
      → POST /api/sessions/:id/open-story → 拿到 streamId → SSE
  → 后端 GameOrchestrator.openStory()
      → PhaseManager.canPerformAction(OPEN_STORY) ✓
         （前置条件：protagonist 非空、openingDone=false、phase=CHARACTER_SETTING）
      → advancePhase: phase → STORY_PLAY, openingDone=true
      → subState = LLM_STREAMING
      → _runLlmFlow(session, STORY_OPENING, '', streamId)
          → InputAssembler: systemInstruction=NARRATIVE_OPENING_INSTRUCTION
          → 世界观+主角设定注入 user message
          → LLM 返回开幕 JSON（narration + locations/npcs/items/options）
      → OutputProcessor: 分支='NARRATIVE'
          → EntityUpdater.applyNarrative():
              → narration → chatRecord
              → locations[] → session.locations (upsert by name)
              → npcs[] → session.npcs
              → items[] → session.inventory
              → hp/san → session.protagonist 文本替换
              → options[] → session.optionBuffer
      → subState = AWAITING_INPUT
      
  → 前端接收 narration + 更新地点/NPC/物品侧边栏 + 渲染选项按钮
```

### 阶段 4：跑团交互循环 (STORY_PLAY)

这是游玩的核心循环，有两种路径：

#### 路径 A：普通叙事（NARRATION_I）

```
玩家输入行动 / 点击选项按钮
  → OptionResolver.resolve(userText, optionBuffer)
      → 若输入为"选项 A"→ 映射为完整行动文本
      → 若输入为"选项 A 和 C"→ 拼接两个选项文本
      → 否则保留原始输入
  → 写入 chatRecord: {role:'player', type:'prompt', content}
  → FlowType → NARRATION_I
  → InputAssembler._buildNarrationIMessages():
      → system message = NARRATION_I_INSTRUCTION
      → NecessarySettingsBuilder.build() → 注入世界观/主角/地点/NPC/物品
      → ConversationHistoryBuilder.formatChatRecord() → 注入历史对话
      → 当前用户输入注入
  → LLM 返回 JSON
      → 若格式 A（无 dice）:
          → OutputProcessor → NARRATIVE 分支
          → EntityUpdater 更新实体状态
          → 前端渲染 narration + 更新侧边栏 + 渲染新选项
      → 若格式 B（有 dice）→ 进入骰子流程
```

#### 路径 B：骰子判定流程

```
LLM 返回: { "narration": "你需要过一个侦查检定【侦查】",
            "dice": { "skill_name":"侦查", "skill_point":60, 
                       "notation":"1d100", "success_rate":50 } }
  
  → OutputProcessor: hasDice() = true → 返回分支 'DICE'
  → _handleDiceBranch():
      → subState = DICE_PENDING
      → session.pendingDiceFlow = { diceNotation, pendingRaw, rollbackChatLen, rollbackDisplayLen }
      → 返回 { branch:'DICE_AWAITING', diceNotation:'1d100' }
      → emit dice_confirm 事件（前端显示确认/取消按钮）
      → emit done（前端保存状态但保持输入锁定）

【玩家点击"确定"】
  → _confirmDice()
      → POST /api/sessions/:id/dice-confirm → streamId → SSE
  → 后端 GameOrchestrator.confirmDice()
      → _executeDice():
          → 将 pending 的 narration + 【】数据正式写入 chatRecord
          → DiceService.parseNotation('1d100') → [{count:1, sides:100}]
          → DiceService.rollAll() → crypto.randomInt(1,101) → [73]
          → DiceService.formatSystemMessage([73]) → "【系统投掷结果】73"
          → emit system 事件 + 写入 chatRecord
          → HistorySummarizer.checkAndRun()（可能触发摘要）
          → InputAssembler.assemble(NARRATION_II, session)
              → systemInstruction = NARRATION_II_INSTRUCTION
              → 注入投掷结果作为 user 消息
          → LLM 返回掷骰后续叙事 JSON
          → OutputProcessor: 可能再次进入 DICE 分支（连环判定）
          → 最终 subState = AWAITING_INPUT
          
  → 前端接收: system 消息显示投掷结果 → narration 渲染 → 选项更新 → 输入解锁

【玩家点击"取消"】
  → _cancelDice()
      → POST /api/sessions/:id/dice-cancel
  → 后端 cancelDice():
      → chatRecord 回滚到 pending 前的长度
      → displayLog 回滚
      → pendingDiceFlow = null
      → subState = AWAITING_INPUT
      → 推送"已取消掷骰判定，请重新选择行动。"
```

#### 路径 C：自动记忆摘要

```
每次 handleMessage 完成后，自动调用 HistorySummarizer.checkAndRun():
  → ConversationHistoryBuilder.estimateCharCount(chatRecord, excludeLatest=true)
  → 若累计 > 8000 字符:
      → subState = SUMMARIZING
      → InputAssembler.assemble(HISTORY_SUMMARY, session)
      → LLM 调用 → 返回 { "summary": "300字以内剧情总结..." }
      → EntityUpdater.applySummary():
          → 保留 chatRecord 最新 2 条
          → 其余替换为一条 SUMMARY 类型条目
      → emit system "系统正在更新记忆" → "更新完成"
      → subState = AWAITING_INPUT（若过程中出错，恢复时自动重置）
```

### 阶段 5：会话中断与恢复

```
玩家关闭浏览器
  → 每次操作后自动 sessionStore.saveSession() → IndexedDB 全量保存
  → URL hash 记录 sessionId

玩家再次打开
  → _init() 从 hash 或 localStorage 恢复 sessionId
  → sessionStore.getSession(id) 
      → 从 IndexedDB 读取
      → 自动重置 LLM_STREAMING / SUMMARIZING 为 AWAITING_INPUT
      → pendingDiceFlow 设为 null（防中间态残留）
  → _restoreUI()
      → 遍历 displayLog[] 重建全部消息
      → 若 subState=DICE_PENDING → 重新渲染确认按钮
      → 若 displayLog 为空 → 从 setupHistory + chatRecord + pendingDiceFlow 
         实时重建（兼容旧数据）
  → _updateUI() → 恢复侧边栏所有状态
```

## 五、关键设计决策总结

| 设计要点 | 实现方式 |
|:---|:---|
| **无状态后端** | 每次请求携带完整 session，处理完写回 SQLite，可水平扩展 |
| **规则与叙事分离** | LLM 提供叙事但不掌控结果，骰子由 `crypto.randomInt` 真随机 |
| **结构化输出** | LLM 必须输出 JSON（`response_format:json_object`），后端解析后分发到不同分支 |
| **JSON 解析容错** | 一次自动重试 + raw 兜底 + `【】`内容保底存储，确保不丢失数据 |
| **双格式渲染** | `plainText` 模拟打字机流式效果，`html` 最终渲染金色分割线样式 |
| **独立实体追踪** | locations/npcs/inventory 存在 session 中，不受 LLM 上下文遗忘影响 |
| **自动记忆管理** | 8000 字符阈值触发 `HISTORY_SUMMARY`，摘要替换旧对话保留最新 2 条 |
| **骰子分支循环** | `_handleDiceBranch` → `confirmDice` → `_executeDice` → 可能再次 `NARRATION_II` 含 dice → 支持连环判定 |
| **会话可靠恢复** | IndexedDB + SQLite 双存储，恢复时自动重置瞬时状态，兼容旧数据重建显示日志 |
| **前端不处理逻辑** | 所有游戏判定在后端，前端只负责 UI 展示和会话缓存 |

以上就是该项目的完整代码框架说明。几个核心要点概括：

1. **架构是"前端薄、后端厚"**：前端只有 UI 渲染和 IndexedDB 缓存，所有游戏逻辑（状态机、输入组装、LLM调用、JSON解析、实体更新、骰子判定、记忆总结）全在后端
2. **无状态设计很巧妙**：后端不存内存状态，每次请求前端携带完整 session 快照，后端处理完写回 SQLite——这意味着可以随时重启后端而不会丢失游戏进度
3. **LLM 输出走的是严格的"JSON → 解析 → 分支路由"流水线**：不是放任 LLM 自由文本，而是强制 JSON 格式，然后由 `OutputProcessor` 分发到 DICE / NARRATIVE / SETUP / SUMMARY 四个分支
4. **骰子判定支持连环循环**：NARRATION_I → DICE 分支 → 用户确认 → NARRATION_II → 可能再次 DICE → ...，直到不再需要判定
5. **会话恢复有容错设计**：瞬时状态（LLM_STREAMING / SUMMARIZING）在恢复时自动重置为 AWAITING_INPUT，即使浏览器崩溃也不会卡死
